# Slack アカウント申請自動化システム 設計書

## 1. システムアーキテクチャ

### 1.1. 全体構成

```
┌─────────────┐
│   Slack     │
│   Workspace │
└──────┬──────┘

│  ┌───────────────────────────┐  │
│  │  slack_bot_handler()     │  │
│  │  - 署名検証               │  │
│  │  - リトライ対策           │  │
```

### 1.2. データフロー

1. **イベント受信**
   - Slackから `app_mention` イベントを受信
   - HTTPヘッダーでリトライ検出
   - 署名検証でリクエストの正当性を確認

2. **問合せ者の特定**
   - `event['user']` からUser IDを取得
   - `users.info` APIを使用してユーザー名（表示名、実名、またはUser ID）を取得して `inquirer_name` として使用（`users:read`スコープが必要）

3. **文字数チェック**
   - メッセージ本文が1000文字以内であることを確認（コスト削減のため）
   - 1000文字を超える場合はエラーメッセージを返して処理を中断

4. **モード判定（課題管理 / 問合せ管理）**
   - メンション除去・`normalize_slack_mailto_links()` によるメール正規化のあと、テキストの**1行目のみ**を対象にキーワードが含まれるか判定して `is_issue` フラグを生成
   - キーワード: `【課題事項】` `【課題】` `【SEJ案件課題】` `[課題事項]` `[課題]` `[SEJ案件課題]` `課題：` `課題事項：`
   - 1行目にキーワードがある場合 → `is_issue = True`（課題管理モード）
   - 1行目にキーワードがない場合 → `is_issue = False`（問合せ管理モード、従来動作）
   - ※ 本文中に「【課題】」等が記述されている場合の誤検知を防ぐため、先頭行のみを判定対象とする

5. **情報抽出**
   - Vertex AI (Gemini) にテキストと問合せ者名を送信して構造化データを抽出
   - `is_issue` に応じてプロンプトと出力JSON構造を切り替え
   - JSON配列形式で複数件を取得（1件のメッセージに複数の依頼・課題がある場合に対応）

6. **データ保存**
   - バリデーション（配列が空でないこと。問合せ管理モードかつタグに「アカウント管理」が含まれる場合のみ `target_email` が必須。ただし1行目に `メールアドレス無し` `メールアドレスは不要` `メールアドレスは有りません` `メールアドレスはありません` `アカウント管理ではない` `アカウント管理ではありません` のいずれかがあればスキップ）
   - `is_issue` に応じて書き込み先ワークシートと列構成を切り替えてスプレッドシートに追記
   - タイムスタンプ付きで記録

7. **レスポンス送信**
   - 処理結果をSlackスレッドに返信
   - 課題管理成功時: 「ご報告いただいた課題について、以下の課題管理シートに{件数}件登録しました。内容をご確認の上、必要に応じて担当者のアサインをお願いします。」+ 課題タイトル・カテゴリ/タグ・優先度などを表示
   - 問合せ管理成功時: 「お問合せ頂いた内容について、以下の通りスプレッドシートに{件数}件登録しました。認識相違が無いかご確認ください。」+ 対象者・メールアドレス・タグなどを表示
   - 失敗時: 適切なエラーメッセージを送信

## 2. 関数設計

### 2.1. `slack_bot_handler(request: Request) -> tuple[str, int]`

**役割**: Cloud Functionsのエントリーポイント

**処理フロー**:
1. リトライ検出（`X-Slack-Retry-Num` ヘッダーチェック）
2. 署名検証（`SLACK_SIGNING_SECRET` を使用）
3. URL検証（`url_verification` イベント対応）
4. イベントタイプ判定（`event_callback` のみ処理）
5. メンションイベント処理
   - User ID取得とユーザー名取得（問合せ者の特定、`users:read`スコープが必要）
   - 文字数チェック（1000文字以内）
   - メンション除去後に `normalize_slack_mailto_links()` でメールの Slack 自動リンク表記を正規化
   - **課題管理キーワード判定**（`【課題事項】` `【課題】` `【SEJ案件課題】` → `is_issue` フラグ生成）
   - Geminiで情報抽出（`is_issue` を引数として渡す）
   - バリデーション（問合せ管理モードかつタグに「アカウント管理」が含まれる場合のみメールアドレス必須）
   - スプレッドシートに書き込み（`is_issue` を引数として渡す、複数件対応）
   - Slackに返信（`is_issue` に応じてメッセージを切り替え）
6. エラーハンドリング

**戻り値**: `(レスポンス本文, HTTPステータスコード)`

### 2.2. `extract_info_with_gemini(text: str, inquirer_name: str, is_issue: bool = False) -> List[Dict[str, Any]]`

**役割**: Vertex AIを使用した情報抽出

**処理内容**:
- Vertex AI (Gemini 2.5 Flash Lite) の初期化
- `is_issue` に応じてプロンプトを切り替え
  - **`is_issue=False`（問合せ管理）**: 問合せ者名をコンテキストとして埋め込み、`target_name` / `target_email` / `tags` / `details` / `due_date` / `order_number` / `order_name` を抽出
  - **`is_issue=True`（課題管理）**: `issue_title` / `issue_summary` / `tags` / `priority` / `due_date` を抽出
- API呼び出し（Temperature: 0, JSONモード強制）
- レスポンスパース（コードブロック除去）
- 期日の正規化（`due_date` フィールドに対して `normalize_due_date()` を適用、両モード共通）
- 配列形式で抽出データのリストを返却（複数依頼・複数課題に対応）

**引数**:
- `text`: 抽出対象のテキスト
- `inquirer_name`: 問合せ者のユーザー名
- `is_issue`: 課題管理モードの場合 `True`（デフォルト: `False`）

**戻り値（問合せ管理）**: `[{"target_name": str|null, "target_email": str|null, "tags": List[str], "details": str|null, "due_date": str|null, "order_number": str|null, "order_name": str|null}, ...]`

**戻り値（課題管理）**: `[{"issue_title": str, "issue_summary": str, "tags": List[str], "priority": str|null, "due_date": str|null}, ...]`

**例外処理**: JSONパースエラー、API呼び出しエラーをキャッチして再スロー

### 2.3. `write_to_spreadsheet(inquirer_name: str, extracted_data_list: List[Dict[str, Any]], slack_url: str, is_issue: bool = False) -> tuple[bool, List[int], List[int]]`

**役割**: Googleスプレッドシートへのデータ書き込み（複数件対応）

**処理内容**:
- ADC (Application Default Credentials) で認証
- `is_issue` に応じてワークシートと列構成を切り替え
  - **`is_issue=False`（問合せ管理）**: ワークシート「問合せリスト」、書き込み範囲 `A:P`（16列）
  - **`is_issue=True`（課題管理）**: ワークシート「課題管理」、書き込み範囲 `A:L`（12列）
- 問合せ（課題）Noを生成（対象シートの既存の最大値+1）
- タイムスタンプ生成（JST）
- リストの各要素を行として最終行に追記
- 新規登録時のみ進捗ステータスに `1.未着手` を自動設定（以降の更新はスプレッドシート上の手作業を想定）

**引数**:
- `inquirer_name`: 問合せ（起票）者のユーザー名
- `extracted_data_list`: 抽出された情報のリスト
- `slack_url`: 問合せ元のSlack URL
- `is_issue`: 課題管理モードの場合 `True`（デフォルト: `False`）

**戻り値**: `(成功時True, 書き込んだ行番号のリスト, 書き込んだNo（問合せNo/課題No）のリスト)`

**前提条件**: 対象ワークシートの1-2行目にヘッダーが存在すること

**日付型の扱い（課題管理モードのみ）**:
- B列（起票日時）・J列（目標解決日）は、文字列ではなく Google Sheets のシリアル値（数値）として書き込む
- シリアル値はエポック（1899-12-30）からの経過日数（時刻は小数部分）
- これにより、スプレッドシート側でセルの書式を「日時」「日付」に設定すれば正しく表示される
- 実装: `_datetime_to_sheets_serial()` / `_date_str_to_sheets_serial()` の2つのヘルパー関数を使用
- 目標解決日が `YYYY-MM-DD` 形式以外（相対表現が残った場合など）は文字列フォールバック

### 2.4. `normalize_due_date(due_date_str: Optional[str]) -> Optional[str]`

**役割**: 期日文字列を正規化する（相対的な日付表現を実際の日付に変換）

**処理内容**:
- 既にYYYY-MM-DD形式の場合はそのまま返す
- 相対的な表現（「○月中」「○月末」「今月末」「来月」「来週」「今週末」「○日後」など）を実際の日付に変換
- 全角・半角数字の両方に対応
- 変換できない場合は元の文字列を返す

**戻り値**: 正規化された日付文字列（YYYY-MM-DD形式）、変換できない場合は元の文字列

### 2.5. `normalize_slack_mailto_links(text: str) -> str`

**役割**: Slack Event API が返すメールアドレスの自動リンク表記を、Gemini に渡しやすい平文に戻す

**処理内容**:
- パターン例: `<mailto:user@example.com|user@example.com>` および `<mailto:user@example.com>` を `user@example.com` に置換
- メンション除去の直後、`extract_info_with_gemini` 呼び出し前に適用する

**戻り値**: 正規化後の文字列

### 2.6. `send_slack_reply(channel, thread_ts, message) -> None`

**役割**: Slackスレッドへの返信送信

**処理内容**:
- Slack WebClientの初期化（`SLACK_BOT_TOKEN` を使用）
- `chat.postMessage` API呼び出し
- スレッドに返信（`thread_ts` を使用）

**例外処理**: API呼び出しエラーをキャッチして再スロー

## 3. セキュリティ設計

### 3.1. 認証・認可

- **Slack署名検証**: `SLACK_SIGNING_SECRET` を使用してリクエストの正当性を検証
- **問合せ者の特定**: `event['user']` からUser IDを取得し、`users.info` APIを使用してユーザー名（表示名、実名、またはUser ID）を取得（`users:read`スコープが必要）
- **Slack URL生成**: `team.info` APIを使用してworkspace名を取得し、正しいメッセージURL形式を生成（`team:read`スコープが必要）
- **Google認証**: ADC (Application Default Credentials) を使用
  - Cloud Functionsのサービスアカウントに必要な権限を付与
  - スプレッドシートへの読み書き権限が必要

### 3.2. リトライ対策

- `X-Slack-Retry-Num` ヘッダーが存在する場合、処理をスキップして即座に200を返す
- 二重書き込みを防止

### 3.3. エラーハンドリング

- すべての関数で例外処理を実装
- Cloud Loggingにエラーログを出力
- ユーザーには適切なエラーメッセージを返信

## 4. データ構造

### 4.1. スプレッドシート構造

#### 問合せリスト（問合せ管理モード: `is_issue=False`）

| 列 | 項目名 | 説明 |
|---|---|---|
| A | **問合せNo** | 連番（自動採番） |
| B | タイムスタンプ | `YYYY-MM-DD HH:MM:SS` 形式 |
| C | **問合せ者** | 問合せ者名（`users.info` で取得した表示名・実名等、`inquirer_name`） |
| D | **概要・詳細** | `details`（空欄可） |
| E | **問合せ元Slack URL** | メッセージへのリンク（`https://{workspace}.slack.com/archives/{channel}/p{ts}`形式） |
| F | **タグ1** | 問い合わせの属性を表すタグ（例：アカウント管理、スラック、課題など） |
| G | **タグ2** | 問い合わせの属性を表すタグ（空欄可） |
| H | **タグ3** | 問い合わせの属性を表すタグ（空欄可） |
| I | **タグ4** | 問い合わせの属性を表すタグ（空欄可） |
| J | **タグ5** | 問い合わせの属性を表すタグ（空欄可） |
| K | **【対象】氏名** | `target_name`（空欄可） |
| L | **【対象】Email** | `target_email`（タグに「アカウント管理」が含まれる場合のみ必須、それ以外は空欄可） |
| M | 対応期日 | `due_date`（YYYY-MM-DD形式、空欄可。相対的な表現は正規化される） |
| N | オーダ番号 | `order_number`（990000～999999の6桁、または数字2桁+JP+数字6桁形式、該当しない場合は空欄） |
| O | オーダ名 | `order_name`（オーダ番号の後に続くテキスト、オーダ番号がない場合は空欄） |
| P | **進捗ステータス** | 本システムが新規行を追加するときのみ `1.未着手` を自動設定。以降の更新は運用で手入力 |

#### 課題管理（課題管理モード: `is_issue=True`）

| 列 | 項目名 | 説明 |
|---|---|---|
| A | **課題No** | 連番（自動採番、問合せNoと同じロジック） |
| B | 起票日時 | `YYYY-MM-DD HH:MM:SS` 形式 |
| C | **起票者** | 起票者名（`users.info` で取得した表示名・実名等） |
| D | **Slack URL** | 元メッセージへのリンク |
| E | **課題タイトル** | `issue_title`（20文字以内の要約） |
| F | **課題概要** | `issue_summary`（状況・影響・背景を含む詳細サマリー） |
| G | **カテゴリ/タグ** | `tags` をカンマ区切りの文字列に変換して格納（null要素は除外） |
| H | 優先度 | `priority`（High / Mid / Low、不明な場合は空欄） |
| I | 担当者 | 空欄（スプレッドシート上で手入力） |
| J | 目標解決日 | `due_date`（YYYY-MM-DD形式または相対表現、空欄可。相対的な表現は正規化される） |
| K | **ステータス** | 新規登録時のみ `1.未着手` を自動設定 |
| L | 最新状況メモ | 空欄（スプレッドシート上で手入力） |

### 4.2. Gemini抽出データ構造

#### 問合せ管理モード（`is_issue=False`）

```json
[
  {
    "target_name": "対象者の氏名またはnull",
    "target_email": "メールアドレスまたはnull",
    "tags": ["タグ1", "タグ2", "タグ3", "タグ4", "タグ5"],
    "details": "概要・詳細またはnull",
    "due_date": "対応期日（YYYY-MM-DD形式、相対表現、またはnull）",
    "order_number": "オーダ番号またはnull",
    "order_name": "オーダ名またはnull"
  }
]
```

**タグについて**:
- 最大5つのタグを設定可能（不足する場合はnullで埋める）
- タグの例：「アカウント管理」「アカウント新規申請登録」「スラック」「課題」「作業依頼」など
- **重要**: 問い合わせ内に所属を表す情報が含まれる場合は、その所属情報もタグに含める

#### 課題管理モード（`is_issue=True`）

```json
[
  {
    "issue_title": "課題タイトル（20文字以内）",
    "issue_summary": "課題の概要（詳細なサマリー）",
    "tags": ["タグ1", "タグ2"],
    "priority": "High/Mid/Lowまたはnull",
    "due_date": "目標解決日（YYYY-MM-DD形式、相対表現、またはnull）"
  }
]
```

**共通事項（期日について）**:
- 明示的な日付（例：「2024-01-31」）の場合はYYYY-MM-DD形式で返す
- 相対的な表現（例：「１月中」「来月末」「今週末」など）の場合は、そのままの表現で返す
- 抽出後、両モード共通で `normalize_due_date()` 関数により相対的な表現を実際の日付（YYYY-MM-DD形式）に変換する

**注意**: 1つのメッセージに複数の依頼・課題がある場合は、配列に複数の要素が含まれる。

## 5. 環境変数

| 変数名 | 説明 | 必須 |
|---|---|---|
| `SLACK_BOT_TOKEN` | Slack Bot User OAuth Token | 必須 |
| `SLACK_SIGNING_SECRET` | Slack Signing Secret | 必須 |

## 6. 依存ライブラリ

- `functions-framework>=3.5.0`: Cloud Functionsエントリーポイント

## 7. ロギング設計

- **ログレベル**: INFO以上
- **ログ出力先**: Cloud Logging（自動）
- **ログ内容**:
  - メンション受信
  - 抽出されたデータ
  - スプレッドシート書き込み成功/失敗
  - Slack返信成功/失敗
  - エラー詳細

## 8. パフォーマンス考慮事項

- **Vertex AI**: `gemini-2.5-flash-lite` を使用（低レイテンシ・低コスト、Gemini 1.5は廃止済み）
- **Temperature**: 0に設定（一貫性のある結果）
- **JSONモード**: 強制してパース処理を簡素化
- **非同期処理**: 現在は同期処理だが、将来的に非同期化可能
