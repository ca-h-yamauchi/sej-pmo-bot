# Slack アカウント申請自動化システム 実装要件定義書

あなたはGoogle Cloud Platform (GCP) およびPython開発のエキスパートです。
以下の要件に基づき、Slackからのメンションを受け取り、Geminiで情報を抽出してGoogleスプレッドシートに転記する「サーバーレス・ボット」のPythonコード（Cloud Functions用）を作成してください。

## 1. システム概要
* **トリガー**: Slackでボットに対してメンションが飛んでくる（例：「@Bot 佐藤太郎さんのアカウント作成をお願い。営業部、sato@example.com」）。
* **処理**:
    1.  Slack Event APIからCloud Functionsがリクエストを受信。
    2.  Slack APIで送信者の表示名を取得し、問合せ者（起票者）として記録。
    3.  メンション除去後、Slackのメール自動リンク（`<mailto:...>`）を平文に正規化。
    4.  メッセージに課題管理キーワード（`【課題事項】` `【課題】` `【SEJ案件課題】`）が含まれるかで**課題管理モード**と**問合せ管理モード**を分岐。
    5.  受信テキストと問合せ者名をVertex AI (Gemini 2.5 Flash Lite) に送り、モードに応じたプロンプトで構造化データ（JSON配列）として抽出。
    6.  抽出したデータ（複数件対応）をモードに応じたワークシートの末尾に行として追加（1件ずつ行を分けて登録）。
    7.  Slackのスレッドにモードに応じた処理結果を返信。

## 2. 技術スタック
* **Runtime**: Python 3.11+
* **Platform**: Google Cloud Functions (第2世代 / HTTP Trigger)
* **AI Model**: Vertex AI `gemini-2.5-flash-lite` (低レイテンシ・低コストのため選択)
* **Libraries**:
    * `functions-framework`: Cloud Functionsのエントリーポイント用
    * `slack-sdk`: Slack API操作用
    * `google-cloud-aiplatform`: Vertex AI操作用
    * `gspread`: Google Sheets操作用 (および `google-auth`)

## 3. 詳細実装要件

### 3.1. エントリーポイントとセキュリティ (`main.py`)
* 関数名: `slack_bot_handler`
* **問合せ者の特定**:
    * `event['user']` からUser IDを取得し、`users.info` APIを使用してユーザー名（表示名、実名、またはUser ID）を取得して `inquirer_name` とする（`users:read`スコープが必要）。
    * AIのプロンプトに埋め込み、スプレッドシートの「問合せ者」/「起票者」列に記録する。
* **Slackリトライ対策 (重要)**:
    * HTTPヘッダー `X-Slack-Retry-Num` が存在する場合、処理を行わずに即座にHTTP 200を返して終了する（二重書き込み防止）。
* **署名検証**:
    * `SLACK_SIGNING_SECRET` を使用して、リクエストが正当なSlackからのものか検証する。
* **URL検証 (url_verification)**:
    * Slack APIの仕様に従い、`type` が `url_verification` の場合は `challenge` パラメータをそのまま返す処理を含める。
* **Slackメール表記の正規化**:
    * Event API ではメールアドレスが `<mailto:アドレス|表示>` 等で届くことがある。`normalize_slack_mailto_links()` で Gemini に渡す前に平文のメール表記へ戻す。
* **課題管理 / 問合せ管理の分岐**:
    * メンション除去・メール正規化後のテキストの**1行目のみ**を対象に、以下のキーワードが含まれるか判定して `is_issue` フラグを生成する。
    * キーワード: `【課題事項】` `【課題】` `【SEJ案件課題】` `[課題事項]` `[課題]` `[SEJ案件課題]` `課題：` `課題事項：`
    * 本文中（2行目以降）に「【課題】」等が含まれる場合の誤検知を防ぐため、先頭行のみを判定対象とする。
    * `is_issue` フラグを `extract_info_with_gemini()` および `write_to_spreadsheet()` に引数として渡す。

### 3.2. AI解析ロジック (Vertex AI)
* **モデル**: `gemini-2.5-flash-lite`
* **設定**:
    * Temperature: 0
    * Response Mime Type: `application/json` (JSONモードを強制)
* **プロンプト（`is_issue=False`: 問合せ管理モード）**:
    * **コンテキスト情報**: 問合せ者名（`inquirer_name`）をプロンプトに埋め込み、「私」や「自分」の言及があれば対象者を問合せ者名として扱う。
    * **出力形式**: JSONオブジェクトの**配列（List）**。1つのメッセージに複数の依頼がある場合は分割する。
    * **抽出する情報**:
        * **対象者情報**:
            * `target_name`: 対象者の氏名（「私」の場合は問合せ者名）
            * `target_email`: 対象者のメールアドレス (不明な場合は `null`)。ラベル付き・同一行・隣接行の表記や記号付き表記からも、文脈に応じて見落とさず抽出するようプロンプトで指示する。
        * **タグ情報**:
            * `tags`: 問い合わせの属性を表すタグの配列（最大5つ、不足する場合はnullで埋める）
            * タグの例：「アカウント管理」「アカウント新規申請登録」「スラック」「課題」「作業依頼」など
            * **重要**: 問い合わせ内に所属を表す情報が含まれる場合はその所属情報もタグに含めること。
        * **その他**:
            * `details`: 概要・詳細（不明な場合は `null`）
            * `due_date`: 対応期日（明示的な日付はYYYY-MM-DD形式、相対表現はそのまま、不明な場合は `null`）
        * **オーダ情報**:
            * `order_number`: オーダ番号（990000～999999の6桁、または数字2桁+JP+数字6桁の形式。該当しない場合は `null`）
            * `order_name`: オーダ名（オーダ番号がない場合は `null`）
* **プロンプト（`is_issue=True`: 課題管理モード）**:
    * **コンテキスト情報**: 起票者名（`inquirer_name`）をプロンプトに埋め込む。
    * **出力形式**: JSONオブジェクトの**配列（List）**。1つのメッセージに複数の課題がある場合は分割する。
    * **抽出する情報**:
        * `issue_title`: 課題タイトル（メッセージ内容から核心を20文字以内で端的に要約）
        * `issue_summary`: 課題の概要（状況・影響・背景などを含む詳細なサマリー）
        * `tags`: 対象システム名やバグ/仕様/運用などのカテゴリを表すタグの配列
        * `priority`: 優先度（High / Mid / Low。不明な場合は `null`）
        * `due_date`: 目標解決日（明示的な日付はYYYY-MM-DD形式、相対表現はそのまま、不明な場合は `null`）
* **バリデーション**:
    * **文字数チェック**: メッセージ本文が1000文字以内であることを確認（コスト削減のため）。1000文字を超える場合はエラーメッセージを返す。
    * AIのレスポンスをパースし、配列が空でないことを確認する。
    * **問合せ管理モードのみ**: タグに「アカウント管理」が含まれる場合のみ `target_email` が必須。課題管理モードでは適用しない。
    * ただし、メッセージの1行目に以下のいずれかのキーワードがある場合はバリデーションをスキップしてそのまま登録する: `メールアドレス無し` `メールアドレスは不要` `メールアドレスは有りません` `メールアドレスはありません` `アカウント管理ではない` `アカウント管理ではありません`
    * 必須項目が欠けている場合は、スプレッドシートへの書き込みをスキップし、ユーザーに適切なエラーメッセージを返す。
* **期日の正規化**:
    * Geminiから抽出された期日が相対的な表現（例：「１月中」「来月末」「今週末」など）の場合、`normalize_due_date()`関数で実際の日付（YYYY-MM-DD形式）に変換する（両モード共通）。
    * 対応パターン：「○月中」「○月末」「今月末」「来月」「来週」「今週末」「○日後」など。
    * 全角・半角数字の両方に対応。

### 3.3. スプレッドシート連携
* 認証: ADC (Application Default Credentials) を使用（Cloud Functionsのサービスアカウント権限を利用）。
* 対象シート: 環境変数 `SPREADSHEET_KEY` で指定。
* 書き込み: 1-2行目にヘッダーがある前提で、抽出されたデータリストの各要素を最終行の下に追記する（複数件対応）。
* **書き込み先ワークシートと列構成は `is_issue` フラグで切り替わる**:

    **問合せ管理モード（`is_issue=False`）**: ワークシート「問合せリスト」、書き込み範囲 `A:P`（16列）

    | 列 | 項目名 | 内容 |
    |:--|:--|:--|
    | A | **問合せNo** | 連番（自動採番） |
    | B | タイムスタンプ | `YYYY-MM-DD HH:MM:SS` |
    | C | **問合せ者** | `inquirer_name` |
    | D | **概要・詳細** | `details` |
    | E | **問合せ元Slack URL** | メッセージへのリンク |
    | F～J | **タグ1～5** | 問い合わせの属性タグ（各列に1つ）|
    | K | **【対象】氏名** | `target_name` |
    | L | **【対象】Email** | `target_email` |
    | M | 対応期日 | `due_date` |
    | N | オーダ番号 | `order_number` |
    | O | オーダ名 | `order_name` |
    | P | **進捗ステータス** | 新規登録時のみ `1.未着手` を自動設定 |

    **課題管理モード（`is_issue=True`）**: ワークシート「課題管理」、書き込み範囲 `A:L`（12列）

    | 列 | 項目名 | 内容 |
    |:--|:--|:--|
    | A | **課題No** | 連番（自動採番） |
    | B | 起票日時 | `YYYY-MM-DD HH:MM:SS` |
    | C | **起票者** | `inquirer_name` |
    | D | **Slack URL** | 元メッセージへのリンク |
    | E | **課題タイトル** | `issue_title`（20文字以内） |
    | F | **課題概要** | `issue_summary` |
    | G | **カテゴリ/タグ** | `tags` をカンマ区切り文字列で格納 |
    | H | 優先度 | `priority`（High / Mid / Low）|
    | I | 担当者 | 空欄（手入力） |
    | J | 目標解決日 | `due_date` |
    | K | **ステータス** | 新規登録時のみ `1.未着手` を自動設定 |
    | L | 最新状況メモ | 空欄（手入力） |

### 3.4. Slackへのレスポンス
* **問合せ管理成功時**: 「お問合せ頂いた内容について、以下の通りスプレッドシートに{件数}件登録しました。認識相違が無いかご確認ください。」と返信し、スプレッドシートの範囲リンク（問合せNoを含む `A:P` 行範囲）を表示。各件の対象者・メールアドレス・タグ・対応期日を表示。
* **課題管理成功時**: 「ご報告いただいた課題について、以下の課題管理シートに{件数}件登録しました。内容をご確認の上、必要に応じて担当者のアサインをお願いします。」と返信し、スプレッドシートの範囲リンク（課題Noを含む `A:L` 行範囲）を表示。各件の課題タイトル・カテゴリ/タグ・優先度・目標解決日を表示。
* 文字数超過時: 「お問合せの内容が長すぎます（{文字数}文字）。1000文字以内で再度入力してください。」と返信。
* 抽出失敗時: 「情報を正しく読み取れませんでした。再度入力してください」と返信。
* バリデーションエラー時（問合せ管理のみ）: タグに「アカウント管理」が含まれる場合でメールアドレスがない場合、「アカウント管理の依頼には対象者のメールアドレスが必要です。メールアドレスを含めて再度入力してください」と返信。

## 4. 必要なファイル構成
以下のファイルを作成してください。

1.  `main.py`: 上記ロジックを含むメインコード。
2.  `requirements.txt`: 必要なライブラリとバージョン。
3.  `.env.example`: 必要な環境変数の一覧（`SLACK_BOT_TOKEN`, `SLACK_SIGNING_SECRET`, `SPREADSHEET_KEY`, `PROJECT_ID`, `LOCATION`）。

## 5. コーディングの注意点
* 例外処理（Try-Except）を適切に入れ、エラー時はログ（Cloud Logging）に出力しつつ、Slackには「エラーが発生しました」と返すようにすること。
* コードは可読性を重視し、Type Hinting (型注釈) をつけること。
